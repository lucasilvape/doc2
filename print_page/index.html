    PIPELINES MONGODB¶

    Objetivo¶

    Visão Geral do Projeto

    Este repositório contem pipelines de ETL (Extract, Transform, Load) que extraem dados de collections MongoDB e os inserem em tabelas PostgreSQL.

    Etapa Descrição
    Extração Dados brutos de duas collections MongoDB principais
    Transformação Limpeza, conversões de tipos e validações independentes
    Carga Inserção dos dados processados em tabelas PostgreSQL

    Collections Processadas

    • implementations/infos - Dados do produto/apólice
    • installments/parcelas - Dados de parcelamento

    Modos de Atualização

    Suporte a modos: DIARIO, INCREMENTAL, FULL e DINAMICO (escolha automática)

    Tecnologias e Stack¶

    Os pipelines utilizam um conjunto de tecnologias modernas para garantir performance, confiabilidade e manutenibilidade:

    Stack Principal¶

    Tecnologia Descrição
    Python 3.x Linguagem principal para toda lógica de processamento
    MongoDB Banco de dados NoSQL de origem (collections de produção)
    PostgreSQL Banco de dados relacional de destino (data warehouse)
    Pandas Manipulação e transformação de dados em larga escala
    Joblib Processamento paralelo para otimização de performance
    Pandera Validação rigorosa de schemas e qualidade de dados
    Poetry Gerenciamento de dependências e ambientes virtuais
    Jenkins Orquestração e automação de pipelines CI/CD
    MkDocs Documentação técnica e manuais de usuário

    Características Técnicas¶

    Característica Descrição
    Processamento Paralelo Utilização de múltiplos cores para otimizar performance
    Validação de Schema Garantia de qualidade e consistência dos dados
    Modo Dinâmico Escolha inteligente entre processamento incremental ou completo
    Logs Estruturados Monitoramento detalhado de execução e performance
    Configuração Flexível Suporte a múltiplos ambientes (produção/homologação)

    Fluxo de Execução do Pipeline (main_<produto>.py)¶

    Todos os pipelines seguem o mesmo fluxo de execução padronizado:

    Etapa Descrição Modulo/Função
    1. Configuração de Path Adiciona diretório pai ao sys.path para imports os, sys
    2. Carregamento de Config Carrega argumentos do Jenkins via CLI load_pipeline_args()
    3. Log de Configuração Imprime configurações da execução atual print()
    4. Geração de Base Identifica bilhetes a processar por tipo de atualização gerar_base_para_busca()
    5. Geração de Queries Cria pipelines MongoDB em lotes gerar_querys_buscar_*()
    6. Limpeza de Stage Remove e recria pastas temporárias shutil.rmtree()
    7. Exportação Paralela Exporta dados para CSV usando joblib exportar_dados_csv_info_parcela()
    8. Remoção de Indices Remove indices antes da carga (modo FULL) remover_indices_postgres()
    9. Importação Postgres TRUNCATE/DELETE + COPY paralelo atualizar_tabela_postgres_arquivo()
    10. Criação de Indices Recria indices apos carga (modo FULL) criar_indices()

    Referencia

    Para detalhes sobre load_pipeline_args() e constantes dos produtos, consulte Ferramentas.

    Geração de Base (gerar_base_para_busca)

    Filtra bilhetes por data de criação com dias retroativos configuráveis.

    Processa bilhetes de uma data especifica (formato yyyy-mm-dd)

    Processa apenas bilhetes modificados nos últimos N dias

    Processa todos os bilhetes, independente de modificações

    Escolhe automaticamente entre FULL ou INCREMENTAL baseado no percentual de modificações

    Processamento de Duas Collections Separadas¶

    Campos de identificacao comuns entre as collections:

    Campo (Implementações) Campo (Parcelas) Descrição
    _id _id UUID único do documento
    _created _created Data de criação
    _modified _modified Data de modificação

    Arquitetura de Processamento Independente

    • Cada collection é processada separadamente sem operações de junção
    • Campos UUID são convertidos de Binary para string via converter_bytes_para_uuid()
    • Dados são inseridos em tabelas PostgreSQL distintas (*_infos e *_parcelas)
    • Relacionamento é mantido através de chaves UUID compartilhadas

    Boas Praticas¶

    Performance¶

    Pratica Recomendacao
    Lotes Use lotes configuraveis (QDE_BILHETES)
    Paralelizacao Configure adequadamente (QDE_JOBS)
    Memoria Monitore uso em volumes grandes
    Garbage Collection Use gc.collect() apos deletar DataFrames grandes

    Dica de Performance

    Para volumes grandes de dados, considere ajustar QDE_BILHETES e QDE_JOBS de acordo com os recursos disponíveis.

        DIARIO: Deverá receber uma data no padrão 'yyyy-mm-dd' para buscar os bilhetes daquele dia.
        INCREMENTAL: Considera as emissões nos ultimos dias(qde) , a quantidade de dias deverá ser informada na variável qde_dias_retroativos.
        FULL: Considera todos os bilhetes, independente de modificações.
        DINAMICO: Determina o tipo de atualização com base no percentual de modificações
                Se o percentual de modificações exceder o limite informado na variável "percentual_modificacao_incremental" será retornado a base full,
                caso contrário retornará somente os registros que foram modificados nos ultimos dias, a quantidade de dias deverá respeitar a variável qde_dias_retroativos .

    '''
    INCREMENTAL_CYBER = False

    if not tp_atualizacao in ['DIARIO','INCREMENTAL','FULL','DINAMICO']:
        raise ValueError("tp_atualizacao deve ser 'DIARIO', 'INCREMENTAL', 'FULL' ou 'DINAMICO'.")
    if tp_atualizacao == 'DIARIO' and not data_criacao:
        raise ValueError("Para tp_atualizacao 'DIARIO', deve ser fornecida uma data de criação no formato 'yyyy-mm-dd'. A data deve ser informada na variável data_criacao.")
    if tp_atualizacao == 'DIARIO':
        try:
            data = pd.to_datetime(data_criacao, format='%Y-%m-%d')
        except:
            raise ValueError("data_criacao deve estar no formato 'yyyy-mm-dd'.")

    else:
        #print('FULL CYBER')
        df_implementacao = buscar_bilhetes_por_data_criacao_implementacao(ambiente=ambiente,collection=nm_collection_implementacao)

    if nm_collection_pacelas == 'implementations.installmentsgeneric' and tp_atualizacao == 'DINAMICO':
        INCREMENTAL_CYBER = True
        df_parcelas = buscar_bilhetes_por_data_criacao_parcelas(ambiente=ambiente,collection=nm_collection_pacelas,qde_dias_retroativos=qde_dias_retroativos)
    else:
        #print('FULL CYBER')

        print(f'########################  TIPO ATUALIZACAO DINAMICA -- INCREMENTAL -- PICPAY CYBER  ########################')

        return 'INCREMENTAL' ,  df_implementacao,df_parcelas

        df['data_modificacao'] = pd.to_datetime(df['data_modificacao'])

        print(f'########################  TIPO ATUALIZACAO FULL  ########################')
        return 'FULL',df_implementacao,df_parcelas

        print(f'########################  TIPO ATUALIZACAO DIARIO  ########################')
        df_implementacao = df_implementacao[df_implementacao['data_criacao'] == data_criacao]
        df_parcelas = df_parcelas[df_parcelas['data_criacao'] == data_criacao]
        return 'DIARIO',df_implementacao,df_parcelas

    elif tp_atualizacao == 'INCREMENTAL':
        print(f'########################  TIPO ATUALIZACAO INCREMENTAL  ########################')
        dt_max_imp = max(df_implementacao['data_criacao'])

        return 'INCREMENTAL',df_implementacao,df_parcelas

    def remover_indices_postgres(ambiente: Literal['PRODUCAO','HOMOLOGACAO','ANALYTICS'], nm_tabela: str):
    """
    Remove todos os índices de uma tabela, exceto índices de PK/FK/unique constraints.
    """
    schema = 'public'  # ajuste se necessário
    conn = abrir_conexao_postgres(ambiente=ambiente)

    try:
        with conn.cursor() as cur:
            # Buscar índices que não são constraints
            cur.execute("""
                SELECT indexname
                FROM pg_indexes
                WHERE schemaname = %s
                AND tablename = %s
                AND indexname NOT IN (
                    SELECT conname
                    FROM pg_constraint
                    WHERE conrelid = %s::regclass
                )
            """, (schema, nm_tabela, nm_tabela))

            indexes = cur.fetchall()

            for idx_name, in indexes:
                print(f"Dropping index {idx_name}...")
                cur.execute(sql.SQL("DROP INDEX IF EXISTS {}").format(sql.Identifier(idx_name)))

        conn.commit()
        print("Todos os índices removidos (exceto constraints).")

    finally:
        conn.close()

    def converter_coluna_para_numerico(df, coluna):
    """
    Converte uma coluna de texto com vírgulas (e opcionalmente separadores de milhar)
    para valores numéricos (float). Valores inválidos são convertidos para NaN.

        .str.replace(',', '.', regex=False)        # converte decimal vírgula para ponto
        .pipe(pd.to_numeric, errors='coerce')      # converte para número, inválidos viram NaN
    )

    def formatar_datas_coluna_rapido(df, coluna, nova_coluna):
    formatos = [
        "%Y.%m.%d",                 # 2025.02.01
        "%Y-%m-%d",                 # 2025-02-01
        "%d/%m/%Y",                 # 01/02/2025
        "%m/%d/%y",                 # 02/01/2025
        "%d-%m-%Y",                 # 01-02-2025
        "%Y/%m/%d",                 # 2025/02/01
        "%d.%m.%Y",                 # 01.02.2025
        "%Y.%m.%d %H:%M:%S",        # 2024.06.25 05:02:46
        "%Y-%m-%d %H:%M:%S",        # 2024-06-25 05:02:46

        # Formatos ISO 8601
        "%Y-%m-%dT%H:%M:%S.%fZ",    # 2024-06-25T05:02:46.123Z
        "%Y-%m-%dT%H:%M:%S%z",      # 2024-06-25T05:02:46+0000
        "%Y-%m-%dT%H:%M:%S.%f%z",   # 2024-06-25T05:02:46.123+0000
        "%Y-%m-%dT%H:%M:%S",        # 2024-06-25T05:02:46 (sem timezone)
    ]

    resultado = pd.Series(pd.NaT, index=df.index)

    # --- 1) Tratamento para números tipo "serial Excel"
    mask = resultado.isna()
    valores = df.loc[mask, coluna]

    # Identifica valores que podem ser datas Excel:
    # int, float ou string numérica
    possivel_excel = valores.apply(
        lambda x: (isinstance(x, (int, float)) and not pd.isna(x)) or
                (isinstance(x, str) and x.isdigit())
    )

    # Converte strings numéricas para int/float
    serie_numerica = pd.to_numeric(valores[possivel_excel], errors="coerce")

    # Limite seguro para evitar OutOfBoundsDatetime
    limite_superior = 500000  # dias (~ano 3273)
    excel_mask = serie_numerica.between(0, limite_superior)

    # Converter números Excel -> datetime somente se estiverem na faixa segura
    if excel_mask.any():
        resultado.loc[serie_numerica.index[excel_mask]] = pd.to_datetime(
            serie_numerica[excel_mask], 
            unit="D",
            origin="1899-12-30",
            errors="coerce"
        )

    # --- 2) Testar todos os formatos fornecidos
    for fmt in formatos:
        mask = resultado.isna()
        if not mask.any():
            break
        resultado[mask] = pd.to_datetime(df.loc[mask, coluna], format=fmt, errors='coerce')

    # --- 3) Fallback final do pandas (excelente para valores ISO)
    mask = resultado.isna()
    if mask.any():
        resultado[mask] = pd.to_datetime(df.loc[mask, coluna], errors='coerce')
    try:
        df[nova_coluna] = resultado.dt.normalize()
    except:
        df[nova_coluna] = resultado

    return df

    parser.add_argument("--data_criacao", type=str, default=None, help="Data de criação para busca diária (formato YYYY-MM-DD)")
    parser.add_argument("--ambiente_mongo", type=str, default='ANALYTICS', help="Ambiente MongoDB: PRODUCAO, HOMOLOGACAO ou ANALITICS")

    args = parser.parse_args()

    # Retorna como dicionário
    return {

    Sobre o Projeto¶

    PIPELINES MONGODB¶

    Visão Geral

    Este projeto foi desenvolvido pela Equipe de Dados Kovr para automatizar a extracao, transformação e carga (ETL) de dados de collections MongoDB para tabelas PostgreSQL.

    A solução foi criada para evitar o uso direto da API dos produtos, que causava grande impacto operacional nos sistemas de produção.

    Objetivo¶

    Padronizar e automatizar o processo de migração de dados entre MongoDB e PostgreSQL, garantindo:

    Pilar Descrição
    Consistencia Todos os produtos seguem o mesmo padrão arquitetural, facilitando manutenção e evolução
    Escalabilidade Processamento paralelo e configuravel para grandes volumes de dados
    Qualidade Validação rigorosa com schemas Pandera garantindo integridade dos dados
    Flexibilidade Multiplos modos de atualização: Diario, Incremental, Full e Dinâmico
