    Objetivo¶

    Este repositório contém pipelines de ETL (Extract, Transform, Load) que extraem dados de collections MongoDB e os inserem em tabelas Postgres.

    Propósito: Todos os produtos seguem o mesmo padrão arquitetural:

    • Extração: Dados brutos de duas collections MongoDB principais
    • Collection de implementations/infos (dados do produto/apólice)

    • Collection de installments/parcelas (dados de parcelamento)

    • Transformação: Limpeza, conversões de tipos, validações e merge entre collections

    • Carga: Inserção dos dados processados em tabelas Postgres específicas por produto

    Suporte a modos de atualização: Incremental (baseado em modificações recentes) ou Full, com escolha automática via modo DINÂMICO.

    Como executar¶

    Este trecho é comum a todos os produtos/pipelines deste repositório. Use-o como referência rápida para rodar qualquer pipeline localmente.

    1. Clone o repositório:
    git clone https://github.com/KovrSeguradoraBI/PIPELINES_MONGODB.git
    1. Entre na pasta e instale dependências (Poetry):
    cd PIPELINES_MONGODB
poetry install

    (opcional) criar venv dentro do projeto:

    poetry config virtualenvs.in-project true

    1. Ajuste variáveis de ambiente conforme .env.example.

    2. Exemplo de execução (modo DINAMICO):

    poetry run python main_<produto>.py --tp_atualizacao DINAMICO

    Observação

    Cada produto tem seu main_<produto>.py.

    Arquitetura Geral dos Pipelines¶

    Visão Geral¶

    Todos os produtos seguem o mesmo padrão arquitetural de ETL (Extract, Transform, Load):

    • Fonte: MongoDB (produção/homologação) com duas collections principais

    • Destino: Tabela Postgres específica por produto

    • Estrutura: Uma collection de "implementação/infos" + uma collection de "parcelas"

    Fluxo Padrão de Processamento¶

    1. Geração de Base (gerar_base_para_busca)¶

    • Filtra bilhetes por data de criação com dias retroativos configuráveis
    • Suporta 4 modos de atualização:
    • DIARIO: processa bilhetes de uma data específica (formato yyyy-mm-dd)
    • INCREMENTAL: processa apenas bilhetes modificados nos últimos N dias
    • FULL: processa todos os bilhetes, independente de modificações
    • DINAMICO: escolhe automaticamente entre FULL ou INCREMENTAL baseado no percentual de modificações

    2. Geração de Queries¶

    • gerar_querys_buscar_infos(): cria pipelines MongoDB para buscar dados de implementação
    • gerar_querys_buscar_parcelas(): cria pipelines MongoDB para buscar dados de parcelas
    • Divide consultas em lotes configuráveis para otimizar performance

    3. Busca e Transformação (buscar_dados_bilhetes)¶

    • Busca: buscar_infos_bilhetes() + buscar_parcelas()
    • Merge: junção entre infos e parcelas usando chaves UUID específicas
    • Transformação: tratamentos_df() aplica limpeza, conversões de tipos e validações
    • Validação: schema Pandera específico por produto

    4. Processamento ETL (tratamentos_df)¶

    • Strings: limpeza e normalização de campos texto
    • Números: conversão com pd.to_numeric(errors='coerce')
    • Datas: criação de campos derivados padronizados
    • Normalização: renomeação de todas as colunas para minúsculas

    5. Exportação e Carga¶

    • CSV Paralelo: exportação usando joblib com configuração de jobs
    • Carga Postgres: via COPY com políticas por tipo de atualização
    • DELETE: para atualizações incremental/diária
    • TRUNCATE: para atualizações full

    Kovr AP

    Visão Geral¶

    O produto Kovr AP extrai dados de duas collections no MongoDB (uma de "implementação"/"infos" e uma de "parcelas" separada), aplica transformações (limpeza, conversões de tipos, validações) e carrega no Postgres. Suporta atualizações incremental (baseado em modificações recentes) ou full.

    Especificações¶

    • Fonte: MongoDB (produção)
    • Collections: implementations.acidentepessoal (implementação) e implementations.installmentsconsumergeneric (parcelas)
    • Destino: Tabela Postgres innoveo_kovr_ap

    Singularidades¶

    Características Específicas

    • Estrutura em duas collections: informações da implementação em implementations.acidentepessoal e parcelas em implementations.installmentsconsumergeneric — buscadas separadamente e depois combinadas
    • Parcela por linha: as parcelas são "explodidas" (one row per installment) em buscar_parcelas antes do merge; o DataFrame final mantém uma linha por parcela (não agrupa parcelas por bilhete)
    • Merge: a junção entre infos e parcelas é feita em buscar_dados_bilhetes usando left_on='info_InstallmentsUUID' e right_on='InstallmentsUUID' com how='outer' por padrão — preserva registros de ambos os lados (infos sem parcelas e parcelas sem infos)

    def mover_arquivos_diarios(path_arquivos):
"""
Move arquivos CSV do diretório STAGE para o diretório principal após processamento.

Esta função realiza a limpeza da pasta principal e depois move todos
os arquivos CSV processados da pasta STAGE para a pasta principal,
finalizando o ciclo de processamento.

Args:
    path_arquivos (str): Caminho base contendo as pastas principal e STAGE

Examples:
    >>> mover_arquivos_diarios('E:\\ARQUIVOS_MONGO\\kovr_ap')
    2025-09-19 14h 30m 45s Limpando pasta principal
    2025-09-19 14h 30m 46s Movendo 5 arquivos

    # Antes:
    # E:\ARQUIVOS_MONGO\kovr_ap\STAGE\2025-01-01.csv
    # E:\ARQUIVOS_MONGO\kovr_ap\STAGE\2025-01-02.csv

    # Depois:
    # E:\ARQUIVOS_MONGO\kovr_ap\2025-01-01.csv
    # E:\ARQUIVOS_MONGO\kovr_ap\2025-01-02.csv

Note:
    - ETAPA 1: Remove todos os .csv da pasta principal
    - ETAPA 2: Move todos os .csv da STAGE para principal
    - Arquivos são movidos (não copiados) - STAGE fica vazia
    - Logs informativos com contagem de arquivos
    - Operação segura - verifica existência antes de remover
"""
lista_arquivos = glob(os.path.join(path_arquivos,'*.csv'))
print(f' {data_hora()} Limpando pasta principal')
for arq in lista_arquivos:
    if os.path.exists(arq):
        os.remove(arq)


path_stage = os.path.join(path_arquivos,'STAGE')
lista_arquivos_stage = glob(os.path.join(path_stage,'*.csv'))

print(f' {data_hora()} Movendo {len(lista_arquivos_stage)} arquivos')
for arq in lista_arquivos_stage:
    shutil.move(arq,path_arquivos)

    Sobre¶

    PIPELINES MONGODB¶

    Este projeto foi desenvolvido pela Equipe de Dados Kovr para automatizar a extração, transformação e carga (ETL) de dados de collections MongoDB para tabelas Postgres, evitando usar a API dos produtos, o que causava um grande impacto operacional.

    Objetivo¶

    Padronizar e automatizar o processo de migração de dados entre MongoDB e Postgres, garantindo:

    • Consistência: Todos os produtos seguem o mesmo padrão arquitetural
    • Escalabilidade: Processamento paralelo e configurável
    • Qualidade: Validação rigorosa com schemas Pandera
    • Flexibilidade: Múltiplos modos de atualização (Diário, Incremental, Full, Dinâmico)
