    Objetivo¶

    Este repositório contém pipelines de ETL (Extract, Transform, Load) que extraem dados de collections MongoDB e os inserem em tabelas Postgres.

    Propósito: Todos os produtos seguem o mesmo padrão arquitetural:

    • Extração: Dados brutos de duas collections MongoDB principais
    • Collection de implementations/infos (dados do produto/apólice)

    • Collection de installments/parcelas (dados de parcelamento)

    • Transformação: Limpeza, conversões de tipos, validações e merge entre collections

    • Carga: Inserção dos dados processados em tabelas Postgres específicas por produto

    Suporte a modos de atualização: Incremental (baseado em modificações recentes) ou Full, com escolha automática via modo DINÂMICO.

    Como executar¶

    Este trecho é comum a todos os produtos/pipelines deste repositório. Use-o como referência rápida para rodar qualquer pipeline localmente.

    1. Clone o repositório:
    git clone https://github.com/KovrSeguradoraBI/PIPELINES_MONGODB.git
    1. Entre na pasta e instale dependências (Poetry):
    cd PIPELINES_MONGODB
poetry install

    (opcional) criar venv dentro do projeto:

    poetry config virtualenvs.in-project true

    1. Ajuste variáveis de ambiente conforme .env.example.

    2. Exemplo de execução (modo DINAMICO):

    poetry run python main_<produto>.py --tp_atualizacao DINAMICO

    Observação

    Cada produto tem seu main_<produto>.py.

    Arquitetura Geral dos Pipelines¶

    Visão Geral¶

    Todos os produtos seguem o mesmo padrão arquitetural de ETL (Extract, Transform, Load):

    • Fonte: MongoDB (produção/homologação) com duas collections principais

    • Destino: Tabela Postgres específica por produto

    • Estrutura: Uma collection de "implementação/infos" + uma collection de "parcelas"

    Fluxo Padrão de Processamento¶

    1. Geração de Base (gerar_base_para_busca)¶

    • Filtra bilhetes por data de criação com dias retroativos configuráveis
    • Suporta 4 modos de atualização:
    • DIARIO: processa bilhetes de uma data específica (formato yyyy-mm-dd)
    • INCREMENTAL: processa apenas bilhetes modificados nos últimos N dias
    • FULL: processa todos os bilhetes, independente de modificações
    • DINAMICO: escolhe automaticamente entre FULL ou INCREMENTAL baseado no percentual de modificações

    2. Geração de Queries¶

    • gerar_querys_buscar_infos(): cria pipelines MongoDB para buscar dados de implementação
    • gerar_querys_buscar_parcelas(): cria pipelines MongoDB para buscar dados de parcelas
    • Divide consultas em lotes configuráveis para otimizar performance

    3. Busca e Transformação (buscar_dados_bilhetes)¶

    • Busca: buscar_infos_bilhetes() + buscar_parcelas()
    • Merge: junção entre infos e parcelas usando chaves UUID específicas
    • Transformação: tratamentos_df() aplica limpeza, conversões de tipos e validações
    • Validação: schema Pandera específico por produto

    4. Processamento ETL (tratamentos_df)¶

    • Strings: limpeza e normalização de campos texto
    • Números: conversão com pd.to_numeric(errors='coerce')
    • Datas: criação de campos derivados padronizados
    • Normalização: renomeação de todas as colunas para minúsculas

    5. Exportação e Carga¶

    • CSV Paralelo: exportação usando joblib com configuração de jobs
    • Carga Postgres: via COPY com políticas por tipo de atualização
    • DELETE: para atualizações incremental/diária
    • TRUNCATE: para atualizações full

    Legenda do Fluxo:

    • 🔵 Azul: Fonte de dados (MongoDB)
    • 🟢 Verde: Conclusão do pipeline
    • 🟠 Laranja: Pontos de decisão críticos
    • 🔴 Vermelho: Operações de limpeza (DELETE/TRUNCATE)
    • 🟣 Roxo: Processamento inicial e índices
    • ⚫ Cinza: Filtros e seleções de dados
    • 🟤 Marrom: Geração de queries MongoDB
    • 🔷 Ciano: Busca e coleta de dados
    • 🟡 Amarelo: DataFrames intermediários
    • 🌸 Rosa: Operações de junção/merge
    • 🟦 Azul Escuro: Tratamentos e transformações
    • 🟢 Verde Escuro: Validação de schema
    • 🟢 Verde Claro: Exportação CSV
    • 🟨 Lima: Importação para PostgreSQL

    Notas Técnicas:

    • Modo DINÂMICO: Avalia % de modificações para decidir entre INCREMENTAL ou FULL
    • Processamento Paralelo: Busca de bilhetes e parcelas executada simultaneamente
    • Otimização de Performance: Índices removidos durante carga e recriados após
    • Validação Rigorosa: Schema Pandera em modo strict antes da exportação

    Estrutura de Arquivos por Produto¶

    Cada produto segue a mesma estrutura de arquivos:

    produto/
├── main_produto.py                     # Orquestração principal
├── pipeline_produto_funcoes.py         # Lógica de negócio e transformações
└── pipeline_produto_notebook.ipynb     # Notebook para desenvolvimento e testes

    Configurações Padrão¶

    Variáveis Comuns (main_<produto>.py):¶

    AMBIENTE_MONGO = 'PRODUCAO'                 # Ambiente MongoDB
QDE_DIAS_RETROATIVOS = 3                    # Dias para busca incremental
PERCENTUAL_MODIFICACAO_INCREMENTAL = 40     # Limite para modo DINAMICO
TP_ATUALIZACAO = 'DINAMICO'                 # Tipo de atualização
QDE_BILHETES = 5000                         # Lotes por query
QDE_JOBS = 30                               # Paralelização

    Constantes Padrão (pipeline_<produto>_funcoes.py):¶

    ENCODING = 'utf-8'
SEP = '|'
EXTENSAO = 'csv'
PATH = r"E:\ARQUIVOS_MONGO\produto"
BANCO_UTILIZADO_PARA_IMPORTACAO = 'POSTGRES'

    Cálculo Padrão de Valor Tarifário¶

    O cálculo do valor_tarifario se repete para todos os produtos que possuem parcelas:

    # Fórmula padrão aplicada na maioria dos produtos
valor_tarifario = round((valor_parcela / 107.38) * 100, 2)

# Produtos com fórmulas específicas:
# - Metlife Celular: divisão por 1.0738
# - Outros podem ter variações conforme regras de negócio

    Variações por Produto

    Alguns produtos podem ter fórmulas específicas de cálculo tarifário. Consulte a documentação individual de cada produto para detalhes.

    Schema e Validação¶

    Campos Padrão Obrigatórios¶

    Todos os produtos devem implementar os seguintes campos no schema Pandera:

    Campo Tipo Obrigatório Descrição
    uuid_bilhete str Sim Identificador único do bilhete
    installmentsuuid str Sim Identificador da parcela
    valor_parcela float Sim Valor da parcela
    valor_tarifario float Sim Valor tarifário calculado
    transactionid str Recomendado ID da transação
    data_inicio_vigencia pd.Timestamp Sim Data início da vigência
    data_fim_vigencia pd.Timestamp Sim Data fim da vigência
    data_vencimento pd.Timestamp Sim Data vencimento da parcela
    data_pagamento pd.Timestamp Opcional Data do pagamento
    data_cancelamento pd.Timestamp Opcional Data do cancelamento
    data_estorno pd.Timestamp Opcional Data de estorno

    Merge e Junção de Dados¶

    Campos típicos usados como chaves de junção entre collections:

    Campo (Info) Campo (Parcela) Descrição
    info_InstallmentsUUID InstallmentsUUID UUID da parcela (principal chave de junção)
    uuid_bilhete uuid_bilhete UUID do bilhete (identificador secundário)

    Observações sobre Merge

    • As chaves exatas variam por produto - consulte documentação específica
    • Campos UUID são convertidos de Binary para string via converter_bytes_para_uuid()
    • Merge outer preserva registros órfãos (infos sem parcelas e parcelas sem infos)

    Logs e Monitoramento¶

    Logs Padrão¶

    • Quantidade de bilhetes processados
    • Quantidade de arquivos gerados
    • Progresso do processamento paralelo
    • Tipo de atualização executada
    • Percentual de modificações (modo DINAMICO)

    Boas Práticas¶

    Performance¶

    • Use lotes configuráveis (QDE_BILHETES)
    • Configure paralelização adequada (QDE_JOBS)
    • Monitore uso de memória em volumes grandes

    Dica de Performance

    Para volumes grandes de dados, considere ajustar QDE_BILHETES e QDE_JOBS de acordo com os recursos disponíveis.
